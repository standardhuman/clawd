### CLIHub ‚Äî Make MCP 94% Cheaper with CLI Wrappers
- **Source:** https://kanyilmaz.me/2026/02/23/cli-vs-mcp.html (197 pts on HN)
- **Idea:** MCP servers dump entire tool schemas (~15,540 tokens for 84 tools) into every conversation. CLIHub converts MCP servers into CLIs with lazy `--help` loading, reducing session start to ~300 tokens. The agent discovers tool details only when needed. The author explicitly models the approach on OpenClaw's `available_skills` block format. One command converts any MCP server to a CLI.
- **Why now:** Token cost is the #1 barrier to running Claws autonomously (as the Feb 25 "Ask HN" thread showed ‚Äî $75/week for a personal Claw). A 94% reduction in tool schema overhead directly reduces operating costs. For anyone running multiple MCP servers, this is immediate savings.
- **Effort to test:** Low ‚Äî `npm install clihub`, convert existing MCP servers. Worth testing with Brian's Notion and other MCP integrations.

### OpenAI's "Capability Gap" ‚Äî Benedict Evans on Why ChatGPT Lacks PMF
- **Source:** https://www.ben-evans.com/benedictevans/2026/2/19/how-will-openai-compete-nkg2x (189 pts, 217 comments on HN)

### tldraw Moving Tests to Private Repo ‚Äî AI Cloning Defense Creates a New Category
- **Source:** https://github.com/tldraw/tldraw/issues/8082 (via Simon Willison)
- **Idea:** tldraw is moving its test suite to a closed-source repo because "a comprehensive test suite is enough to build a completely fresh implementation of any open source library from scratch, potentially in a different language." This was triggered by Cloudflare porting Next.js to Vite in a week using AI. The implication: open-source projects with commercial models now face an existential threat from AI-powered reimplementation. Test suites are the new blueprints.
- **Why now:** This creates a new consulting niche: "AI-proof open-source strategy." Companies with open-source products need to rethink what stays public. Also relevant for Brian's own work ‚Äî if he builds tools, the test suite might be more valuable than the code.
- **Effort to test:** Low ‚Äî awareness item. But the pattern (AI + tests = instant reimplementation) is a powerful argument for why code ownership matters less than workflow/relationship moats.

### FDM-1 ‚Äî Foundation Model for Computer Use (Trained on Video, Not Screenshots)
- **Source:** https://si.inc/posts/fdm1/ (220 pts on HN)
- **What it does:** SI Inc trained a foundation model on 11 million hours of screen recordings using an inverse dynamics model (IDM) to label actions. Their video encoder compresses ~2 hours of 30fps video into 1M tokens ‚Äî 50x more efficient than prior state-of-the-art, 100x more efficient than OpenAI's encoder. Unlike VLM-based agents that see screenshots, FDM-1 processes continuous video, enabling long-horizon tasks like CAD design, car driving, and web fuzzing.
- **How we'd use it:** This is the next generation of computer-use agents. Not immediately actionable, but important to track. When this technology matures, "watch a pro do it, then replicate" becomes a viable training method for any domain ‚Äî including marine service documentation workflows. The fuzzing demo (finding bugs by exploring states) maps interestingly to QA automation.

### OpenSwarm ‚Äî Multi-Agent Claude CLI Orchestrator for Linear/GitHub
- **Source:** https://github.com/Intrect-io/OpenSwarm (Show HN, 22 pts)
- **What it does:** Orchestrates multiple Claude Code instances as autonomous agents. Picks up Linear issues, runs Worker/Reviewer pair pipelines (worker writes code, reviewer checks it, tester validates, documenter captures), reports to Discord, retains memory via LanceDB vector embeddings. Supports model escalation (start with Haiku, escalate to Sonnet after iteration 3), stuck detection, and PR auto-improvement.
- **How we'd use it:** The Worker/Reviewer pair pipeline pattern is directly applicable to any development workflow. The model escalation strategy (cheap model first, expensive model for hard problems) is a cost-optimization pattern Brian should adopt. Worth studying the architecture even if not using the tool directly.

### LLM Deanonymization at Scale ‚Äî Privacy Threat to Track
- **Source:** https://simonlermen.substack.com/p/large-scale-online-deanonymization (251 pts on HN)
- **What it does:** Researchers demonstrated that LLMs can deanonymize online users by inferring personal attributes from anonymous posts, then searching the web to match identities. Tested across HN, Reddit, LinkedIn, and anonymized interview transcripts. "From a handful of comments, LLMs can infer where you live, what you do, and your interests ‚Äî then search for you on the web."
- **How we'd use it:** Important for Brian's own OPSEC and for any client conversations about AI risks. Also relevant for the "responsible AI deployment" consulting angle. The research paper (arxiv:2602.16800) could be cited in any security-focused Claw deployment guide.

### Google API Keys + Gemini ‚Äî Privilege Escalation by Design
- **Source:** https://trufflesecurity.com/blog/google-api-keys-werent-secrets-but-then-gemini-changed-the-rules (432 pts on HN)

### Agent Safehouse — The Missing Security Layer for Local AI Agents
- **Source:** https://agent-safehouse.dev/ (518 pts, 118 comments on HN)
- **Idea:** A single-file bash script that sandboxes any coding agent (Claude Code, Codex, Amp, Gemini) using macOS's native `sandbox-exec`. Deny-first model: nothing is accessible unless explicitly granted. Your current working directory gets read/write access; SSH keys, other repos, and personal files are kernel-blocked. The creator (eugene1g) spent weeks mapping minimum permissions per agent — auto-updates, keychain integration, clipboard pasting all work inside the sandbox. Ships with an interactive Policy Builder where you can generate custom profiles, and you can even use an LLM to generate a least-privilege profile specific to your machine.
- **Why now:** The Clinejection attack we covered on March 6 showed that AI agents in CI/CD pipelines are a real attack surface. Agent Safehouse addresses the local counterpart: your laptop. An HN commenter explicitly asked about adapting it for OpenClaw. As agent-driven development becomes standard (SWE-CI shows agents handling months-long maintenance), sandboxing becomes table stakes.
- **Effort to test:** Low — `curl` a single script, add shell functions to `.zshrc`, and every Claude Code session runs sandboxed by default. Brian can test this in 15 minutes.

### Literate Programming + AI Agents — Documentation That Maintains Itself
- **Source:** https://silly.business/blog/we-should-revisit-literate-programming-in-the-agent-era/ (210 pts, 118 comments)
- **Idea:** The author argues that the fundamental labor cost that killed literate programming — maintaining two parallel narratives (code and prose) — is exactly what LLMs are best at: translation and summarization. ## 🛠️ Tools & Methods

### Agent Safehouse Shell Functions — Sandboxing as Default
- **Source:** https://agent-safehouse.dev/ (518 pts on HN)
- **What it does:** Drop these into `.zshrc` and every agent invocation is sandboxed automatically:
  ```bash
  safe() { safehouse --add-dirs-ro=~/mywork "$@"; }
  claude() { safe claude --dangerously-skip-permissions "$@"; }
  codex() { safe codex --dangerously-bypass-approvals-and-sandbox "$@"; }
  ```
  The sandbox uses macOS's native kernel enforcement — processes physically cannot read denied paths. Works with Claude Code, Codex, Amp, and Gemini CLI. Includes a web-based Policy Builder for generating custom profiles and supports machine-specific override lists for global dotfiles like `.gitconfig`.
- **How we'd use it:** Immediate adoption for Brian's local Claude Code sessions. The Clinejection lesson was about CI/CD pipeline sandboxing; this solves the laptop side. Combined, they create a defense-in-depth approach for running AI agents. For OpenClaw, the HN discussion about adapting Safehouse's sandbox-exec policies for always-on agents (not just CLI sessions) is worth following — this could become a standard security layer for OpenClaw installations.

### SWE-CI Benchmark — A New Standard for Agent Evaluation
- **Source:** https://arxiv.org/abs/2603.03823 (116 pts on HN)
- **What it does:** The first benchmark that evaluates AI coding agents on sustained codebase maintenance rather than one-shot bug fixing. Each task spans an average of 233 days of evolution history and 71 consecutive commits. Measures both resolution (can the agent fix the issue?) and regression (does the fix break something else?). Claude Opus 4.6 scores 0.71 resolve rate — nearly 3x GPT-5.2 (0.23).
- **How we'd use it:** Two ways. (1) When evaluating whether to add a new model to Brian's agent setup, SWE-CI is now the most relevant benchmark — it tests what agents actually do in practice (long-term maintenance), not just what they can do in a lab (one-shot fixes). (2) The 25% regression finding should inform how Brian structures his agent workflows: every agent commit needs a verification step, and complex changes should include a "check N-1 and N+1 layers" prompt (echoing the March 7 suggestion from "We might all be AI engineers now").

### Blogtato — CLI RSS Reader Synced via Git
- **Source:** https://github.com/kantord/blogtato (66 pts on HN)
- **What it does:** A CLI-based RSS/Atom feed reader inspired by Taskwarrior that syncs state via Git. Read/unread status, bookmarks, and feed subscriptions all live in a Git repo. Terminal-native, keyboard-driven, fits into existing workflows without a browser.
- **How we'd use it:** Could replace Noa's current web_fetch-based RSS scanning workflow. Instead of fetching and parsing RSS XML manually each night, Blogtato could maintain feed state persistently, showing only unread items. The Git sync means the feed state survives between cron runs.

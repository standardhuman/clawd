## 🛠️ Tools & Methods

### Context Mode — 98% Context Window Savings for Claude Code
- **Source:** https://mksg.lu/blog/context-mode / https://github.com/mksglu/claude-context-mode (349 pts)
- **What it does:** MCP server that intercepts tool outputs and routes them through isolated subprocesses. Raw data goes into a local SQLite FTS5 index instead of the context window. The agent gets compact summaries; full data is queryable on demand. Playwright snapshots go from 56 KB to 299 bytes. Git logs from 11.6 KB to 107 bytes.
- **How we'd use it:** Install immediately on Brian's Claude Code setup. Combined with CLIHub (reported Feb 26, reduces tool *definitions* by 94%), this addresses context consumption from both directions. Important caveat from the HN discussion: Context Mode only works for built-in tools and CLI-based workflows — it cannot intercept MCP tool responses (those flow via JSON-RPC directly to the model). For third-party MCPs, the pattern must be applied server-side. Still, for the majority of Claude Code usage (bash, git, file reads, web fetches), this is transformative.

### Verified Spec-Driven Development (VSDD) — Methodology for Agent-Coded Projects
- **Source:** https://gist.github.com/dollspace-gay/d8d3bc3ecf4188df049d7a4726bb2a00 (175 pts)
- **What it does:** A methodology that front-loads specification writing before AI implementation. Write detailed specs → AI implements → verify against spec → iterate. The HN debate reveals a practical synthesis: one commenter describes their workflow as "rough plan → get second/third opinions from various AIs → detailed specsheet → write documentation first → expected outcomes → tests → code → run pipeline and fix." Another: "AI is excellent at reverse-engineering specs from existing code, so you can ask it to reflect iterative changes back into the spec."
- **How we'd use it:** For SailorSkills Pro app development, VSDD's spec-first approach makes sense for core features (marketplace matching, service logging). For experimental features, the "spin up five agents" rapid iteration approach is better. The key insight: use spec-driven for things that must be right (billing, auth, data models), use rapid iteration for things that must be explored (UX, content, integrations). Brian's OpenClaw sub-agents are already the perfect tool for the rapid iteration side.

### Simon Willison's "Interactive Explanations" — Fighting Cognitive Debt
- **Source:** https://simonwillison.net/guides/agentic-engineering-patterns/interactive-explanations/
- **What it does:** New agentic engineering pattern: when AI-generated code becomes a black box you don't fully understand, have the agent build interactive HTML explanations — visual, explorable documents that let you build intuition about how the code works. This prevents "cognitive debt" (the agent-coding equivalent of technical debt — code that works but nobody understands).
- **How we'd use it:** When building complex SailorSkills features, ask the agent to produce an interactive explanation alongside the implementation. Example: the marketplace matching algorithm could come with a visual explorer showing how it scores and ranks providers. This is both a development tool (Brian understands his own system) and a potential user-facing feature (show boat owners why a particular diver was recommended).
